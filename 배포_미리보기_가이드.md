# 수정사항 미리보기 가이드 (배포 전 확인)

> 목적: 수정한 내용이 **실제 사용자에게 보이기 전에**, 나 혼자 먼저 확인하고 최종 수정하기 위함.

---

## 핵심 추천: "연습용 임시 주소"로 먼저 확인

지금 구조는 수정 → 푸시하면 **곧장 진짜 사이트(tickjapan.com)에 반영**됨 → 사용자도 바로 봄.
미리 확인할 틈이 없음.

**추천 방식:**
> 수정본을 '임시 주소'에 먼저 올려 나 혼자 확인 → 괜찮으면 진짜 사이트에 반영

Vercel이 이걸 **무료로 자동** 해줌.

### 절차
1. 수정을 `main`이 아닌 **다른 이름의 브랜치**에 올린다
2. Vercel이 **그 수정본만의 임시 주소**를 자동 생성
   (예: `tickjapan-git-test.vercel.app` — 진짜 사이트와 별개)
3. 나 혼자 그 임시 주소에서 확인
4. 마음에 들면 → `main`에 머지 → 그때 진짜 사이트 반영
5. 별로면 → 브랜치 버리면 됨 (사용자는 아무것도 못 봄)

**비유:** 손님 받는 가게(tickjapan.com)는 그대로 두고, 연습용 가게를 잠깐 차려
먼저 바꿔보고, 좋으면 진짜 가게에 적용.

### 명령어 예시 (나중에 참고)
```bash
# 1) 새 브랜치 만들어 작업
git checkout -b test-수정내용

# 2) 수정 후 커밋·푸시
git add -A
git commit -m "수정 내용"
git push -u origin test-수정내용
# → Vercel이 임시 미리보기 URL 자동 생성 (Vercel 대시보드 또는 GitHub PR에서 확인)

# 3) 확인 후 좋으면 main에 반영
git checkout main
git merge test-수정내용
git push
```

## 확인 대상별 방법 정리

| 무엇을 확인? | 방법 |
|---|---|
| **웹/앱 화면·기능** (대부분의 수정) | 위의 **임시 주소(브랜치 미리보기)** ← 이거면 충분 |
| **앱 전용 동작** (ATT 추적 팝업, AppsFlyer 등, 가끔) | **TestFlight**로 나만 먼저 설치해 확인 |

- 거의 모든 수정은 **임시 주소 하나로 확인 가능**.
- 앱 모습도 웹과 똑같이 보이므로 이걸 주력으로 사용.
- 앱 네이티브 전용 기능만 시뮬레이터/실기기(`npx cap run ios`)나 TestFlight로 별도 확인.

---

## 참고: 정식 출시 단계 (앱)
- **TestFlight**: 앱스토어 정식 출시 전, 지정한 사람만 설치해 베타 테스트.
- **단계적 출시(Phased Release)**: 정식 출시도 7일에 걸쳐 1% → 100% 점진 배포 가능.
